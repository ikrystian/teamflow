1. Opis Ogólny Systemu
TeamFlow to nowoczesna aplikacja internetowa do zarządzania zadaniami i projektami dla zespołów. Inspirowana funkcjonalnością systemu Asana, ma na celu usprawnienie współpracy, centralizację komunikacji i zwiększenie produktywności. Aplikacja została zbudowana przy użyciu nowoczesnych technologii, zapewniających szybkość, responsywność i intuicyjny interfejs użytkownika.

Nazwa projektu: TeamFlow

Główny cel: Stworzenie platformy do efektywnego zarządzania zadaniami, projektami i współpracą w zespole.

Technologie:

Framework: Next.js (React)

Interfejs użytkownika: shadcn/ui (zbudowany na Tailwind CSS i Radix UI)

ORM: Prisma

Baza danych: SQLite

Uwierzytelnianie: NextAuth.js (lub inna biblioteka do obsługi autoryzacji)

2. Główne Moduły i Funkcjonalności
2.1. Użytkownicy i Uwierzytelnianie
Rejestracja: Użytkownicy mogą zakładać nowe konta za pomocą adresu e-mail i hasła.

Logowanie: Bezpieczny system logowania z walidacją danych.

Profile użytkowników:

Imię i nazwisko.

Avatar (możliwość wgrania zdjęcia).

Rola w organizacji (np. Administrator, Użytkownik).

Lista przypisanych zadań widoczna w profilu.

2.2. Zespoły (Teams)
Tworzenie zespołów: Użytkownicy mogą tworzyć zespoły i zapraszać do nich innych członków za pomocą adresu e-mail.

Zarządzanie członkami: Właściciel zespołu może dodawać i usuwać członków oraz zarządzać ich uprawnieniami (w przyszłych wersjach).

Przestrzeń zespołu: Każdy zespół ma swoją dedykowaną przestrzeń roboczą z listą projektów i członków.

2.3. Projekty (Projects)
Tworzenie projektów: Możliwość tworzenia nowych projektów w ramach wybranego zespołu.

Atrybuty projektu:

Nazwa projektu.

Opis.

Właściciel projektu.

Przypisany zespół.

Status (np. W toku, Zakończony, Wstrzymany).

Widoki projektu:

Lista (List View): Prosty widok wszystkich zadań w formie listy.

Tablica (Board View): Widok Kanban z kolumnami reprezentującymi statusy zadań (np. "Do zrobienia", "W trakcie", "Gotowe"). Umożliwia przeciąganie zadań między kolumnami.

Kalendarz (Calendar View): Widok zadań na kalendarzu, z uwzględnieniem ich terminów wykonania (due date).

2.4. Zadania (Tasks)
Tworzenie zadań: Szybkie dodawanie nowych zadań w ramach wybranego projektu.

Atrybuty zadania:

Tytuł: Krótka i zwięzła nazwa zadania.

Osoba odpowiedzialna (Assignee): Możliwość przypisania zadania do jednego członka zespołu.

Termin wykonania (Due Date): Ustawienie daty, do której zadanie powinno zostać ukończone.

Projekt: Przynależność do konkretnego projektu.

Status: Aktualny stan zadania (np. Do zrobienia, W trakcie, Gotowe) - powiązany z kolumnami na tablicy Kanban.

Priorytet: Określenie ważności zadania (np. Niski, Średni, Wysoki).

Opis: Szczegółowy opis zadania z możliwością formatowania tekstu (np. listy, pogrubienie).

Podzadania (Subtasks): Możliwość dzielenia większych zadań na mniejsze, łatwiejsze do zarządzania kroki. Każde podzadanie może być odznaczone jako ukończone.

Komentarze: Sekcja do dyskusji na temat zadania, umożliwiająca komunikację w kontekście konkretnej pracy.

Załączniki: Możliwość dodawania plików do zadania.

3. Planowanie i Raporty
3.1. Planowanie
Moje zadania (My Tasks): Spersonalizowany widok, który agreguje wszystkie zadania przypisane do zalogowanego użytkownika ze wszystkich projektów.

Oś czasu (Timeline - funkcja zaawansowana): Wizualizacja planu projektu w czasie (podobna do wykresu Gantta), pokazująca zależności między zadaniami i czas ich trwania.

Kalendarz zespołu: Wspólny kalendarz dla całego zespołu, pokazujący terminy wykonania wszystkich zadań w projektach zespołu.

3.2. Raporty i Analizy
Dashboard projektu: Podstawowy pulpit nawigacyjny dla każdego projektu, pokazujący:

Liczbę zadań ukończonych vs. wszystkich.

Zadania przeterminowane.

Wykres postępu prac (np. wykres spalania - burndown chart).

Raporty wydajności:

Możliwość filtrowania zadań po osobie odpowiedzialnej, statusie czy dacie.

Podsumowanie liczby zadań ukończonych przez poszczególnych członków zespołu w danym okresie.

4. Architektura Techniczna i Struktura Danych
4.1. Architektura Systemu
Frontend (Next.js):

Renderowanie po stronie serwera (SSR) i generowanie statyczne (SSG) dla zapewnienia optymalnej wydajności i SEO.

Routing oparty na plikach (App Router).

Komponenty UI zbudowane przy użyciu shadcn/ui, co zapewnia spójny wygląd i łatwość customizacji.

Backend (Next.js API Routes):

Logika biznesowa aplikacji zaimplementowana jako endpointy API w folderze app/api.

Obsługa zapytań CRUD (Create, Read, Update, Delete) dla wszystkich zasobów (projekty, zadania, zespoły).

Baza Danych (Prisma + SQLite):

Prisma ORM jako warstwa abstrakcji do interakcji z bazą danych. Zapewnia bezpieczeństwo typów i upraszcza zapytania.

SQLite jako lekka, plikowa baza danych, idealna na start projektu i do celów deweloperskich. W przyszłości możliwa łatwa migracja na np. PostgreSQL.

4.2. Schemat Bazy Danych (Prisma Schema)
Poniżej znajduje się przykładowy schemat bazy danych zdefiniowany w pliku schema.prisma.

Fragment kodu

// This is your Prisma schema file,
// learn more about it in the docs: https://pris.ly/d/prisma-schema

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "sqlite"
  url      = "file:./dev.db"
}

model User {
  id        String    @id @default(cuid())
  name      String?
  email     String    @unique
  password  String
  avatarUrl String?
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  teams     Team[]    @relation("TeamMembers")
  assignedTasks Task[]
}

model Team {
  id        String    @id @default(cuid())
  name      String
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  members   User[]    @relation("TeamMembers")
  projects  Project[]
}

model Project {
  id          String   @id @default(cuid())
  name        String
  description String?
  status      String   @default("In Progress") // e.g., "In Progress", "Completed"
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  teamId      String
  team        Team     @relation(fields: [teamId], references: [id])
  tasks       Task[]
}

model Task {
  id          String    @id @default(cuid())
  title       String
  description String?
  status      String    @default("To Do") // e.g., "To Do", "In Progress", "Done"
  priority    String?   // e.g., "Low", "Medium", "High"
  dueDate     DateTime?
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  projectId   String
  project     Project   @relation(fields: [projectId], references: [id])
  assigneeId  String?
  assignee    User?     @relation(fields: [assigneeId], references: [id])
  subtasks    Subtask[]
  comments    Comment[]
}

model Subtask {
  id        String   @id @default(cuid())
  title     String
  isCompleted Boolean  @default(false)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  taskId    String
  task      Task     @relation(fields: [taskId], references: [id])
}

model Comment {
  id        String   @id @default(cuid())
  content   String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  taskId    String
  task      Task     @relation(fields: [taskId], references: [id])
  authorId  String
  author    User     @relation(fields: [authorId], references: [id])
}
5. Plan Rozwoju (Roadmap)
Faza 1 (MVP - Minimum Viable Product):

Uwierzytelnianie użytkowników.

Tworzenie zespołów i projektów.

Podstawowe zarządzanie zadaniami (tworzenie, edycja, przypisywanie, termin).

Widok listy i tablicy Kanban dla projektów.

Faza 2:

Implementacja komentarzy i podzadań.

Widok kalendarza.

Podstawowy dashboard projektu.

Zaproszenia do zespołów.

Faza 3:

Zaawansowane raporty i analityka.

Powiadomienia (w aplikacji i e-mailowe).

Możliwość dodawania załączników do zadań.

Wyszukiwarka globalna.
