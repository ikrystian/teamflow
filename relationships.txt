TeamFlow - Relacje między elementami systemu

## Struktura bazy danych (Prisma Schema)

### User (Użytkownik)
- id: String (PK)
- name: String?
- email: String (unique)
- password: String
- avatarUrl: String?
- createdAt: DateTime
- updatedAt: DateTime

Relacje:
- teams: Team[] (many-to-many przez "TeamMembers")
- assignedTasks: Task[] (one-to-many) - zadania przypisane do użytkownika
- createdTasks: Task[] (one-to-many) - zadania utworzone przez użytkownika
- comments: Comment[] (one-to-many)
- timeEntries: TimeEntry[] (one-to-many) - wpisy czasu pracy

### Team (Zespół)
- id: String (PK)
- name: String
- createdAt: DateTime
- updatedAt: DateTime

Relacje:
- members: User[] (many-to-many przez "TeamMembers")
- projects: Project[] (one-to-many)

### Project (Projekt)
- id: String (PK)
- name: String
- description: String?
- status: String (default: "In Progress")
- createdAt: DateTime
- updatedAt: DateTime
- teamId: String (FK)

Relacje:
- team: Team (many-to-one)
- tasks: Task[] (one-to-many)

### Task (Zadanie)
- id: String (PK)
- title: String
- description: String?
- status: String (default: "To Do")
- priority: String?
- dueDate: DateTime?
- estimatedHours: Float? (szacowany czas w godzinach)
- createdAt: DateTime
- updatedAt: DateTime
- projectId: String (FK)
- assigneeId: String? (FK)
- createdById: String (FK) - autor zadania

Relacje:
- project: Project (many-to-one)
- assignee: User? (many-to-one)
- createdBy: User (many-to-one) - autor zadania
- subtasks: Subtask[] (one-to-many)
- comments: Comment[] (one-to-many)
- timeEntries: TimeEntry[] (one-to-many) - wpisy czasu pracy

### Subtask (Podzadanie)
- id: String (PK)
- title: String
- isCompleted: Boolean (default: false)
- createdAt: DateTime
- updatedAt: DateTime
- taskId: String (FK)

Relacje:
- task: Task (many-to-one)

### Comment (Komentarz)
- id: String (PK)
- content: String
- createdAt: DateTime
- updatedAt: DateTime
- taskId: String (FK)
- authorId: String (FK)

Relacje:
- task: Task (many-to-one)
- author: User (many-to-one)

### TimeEntry (Wpis czasu pracy)
- id: String (PK)
- hours: Float (czas w godzinach, np. 0.5 = 30 minut)
- description: String? (opcjonalny opis pracy)
- date: DateTime (data wykonania pracy)
- createdAt: DateTime
- updatedAt: DateTime
- taskId: String (FK)
- userId: String (FK)

Relacje:
- task: Task (many-to-one)
- user: User (many-to-one)

## API Endpoints

### Authentication
- POST /api/auth/register - Rejestracja użytkownika
- GET/POST /api/auth/[...nextauth] - NextAuth.js handler

### Teams
- GET /api/teams - Lista zespołów użytkownika
- POST /api/teams - Tworzenie nowego zespołu
- GET /api/teams/[teamId] - Szczegóły pojedynczego zespołu
- PATCH /api/teams/[teamId] - Edycja zespołu (nazwa i członkowie)
- GET /api/teams/[teamId]/members - Lista członków zespołu
- POST /api/teams/[teamId]/members - Dodawanie członka do zespołu
- DELETE /api/teams/[teamId]/members?userId=X - Usuwanie członka z zespołu

### Users
- GET /api/users - Lista wszystkich użytkowników (z opcjonalnym wyszukiwaniem)

### Projects
- GET /api/projects - Lista projektów (z filtrowaniem po teamId)
- POST /api/projects - Tworzenie nowego projektu
- GET /api/projects/[projectId] - Szczegóły pojedynczego projektu (BRAKUJE - powoduje 404)
- GET /api/projects/[projectId]/members - Lista członków projektu

### Tasks
- GET /api/tasks - Lista zadań (z filtrowaniem po projectId, assigneeId)
- POST /api/tasks - Tworzenie nowego zadania (z szacowanym czasem)
- GET /api/tasks/[taskId] - Szczegóły pojedynczego zadania
- PATCH /api/tasks/[taskId] - Edycja zadania (tylko autor lub przypisana osoba)

### Time Tracking
- GET /api/tasks/[taskId]/time-entries - Lista wpisów czasu dla zadania
- POST /api/tasks/[taskId]/time-entries - Dodawanie wpisu czasu
- DELETE /api/tasks/[taskId]/time-entries?entryId=X - Usuwanie wpisu czasu

## Przepływ danych

1. Użytkownik loguje się przez NextAuth.js
2. Session zawiera user.id używane w API calls
3. API routes sprawdzają autoryzację przez getServerSession
4. Prisma ORM wykonuje zapytania do SQLite
5. Komponenty React pobierają dane przez fetch() do API endpoints
6. Stan aplikacji zarządzany przez useState/useEffect

## Bezpieczeństwo

- Wszystkie API routes wymagają autoryzacji
- Użytkownicy widzą tylko swoje zespoły/projekty/zadania
- Hasła hashowane przez bcryptjs
- Session zarządzane przez NextAuth.js JWT

## Problemy do rozwiązania

### 404 przy kliknięciu w projekt
**Problem**: Po kliknięciu w projekt na liście projektów użytkownik otrzymuje błąd 404.

**Przyczyna**:
- Brakuje strony szczegółów projektu: `src/app/dashboard/projects/[projectId]/page.tsx`
- Brakuje API endpoint dla pojedynczego projektu: `src/app/api/projects/[projectId]/route.ts`
- Link w ProjectsContent.tsx kieruje na `/dashboard/projects/${project.id}` ale strona nie istnieje

**Rozwiązanie** ✅:
1. ✅ Utworzono API endpoint GET /api/projects/[projectId] - obsługuje GET, PATCH, DELETE
2. ✅ Utworzono stronę src/app/dashboard/projects/[projectId]/page.tsx
3. ✅ Utworzono komponent ProjectDetailsContent do wyświetlania szczegółów projektu
4. ✅ Utworzono komponent CreateTaskDialog dla projektów
5. ✅ Aplikacja działa bez błędów na localhost:3001

**Funkcjonalności**:

   - Sprawdzanie czy użytkownik nie jest już członkiem
   - Zwraca zaktualizowaną listę członków

3. **DELETE /api/teams/[teamId]/members?userId=X** - Usuwanie członka z zespołu
   - Walidacja członkostwa w zespole
   - Sprawdzanie czy użytkownik jest członkiem zespołu
   - Zabezpieczenie przed usunięciem ostatniego członka
   - Zwraca zaktualizowaną listę członków

4. **PATCH /api/teams/[teamId]** - Rozszerzona edycja zespołu
   - Aktualizacja nazwy zespołu (jak wcześniej)
   - Opcjonalna aktualizacja członków przez memberIds[]
   - Walidacja wszystkich ID użytkowników
   - Zabezpieczenie przed pustym zespołem
   - Atomowa operacja (wszystko lub nic)

5. **EditTeamDialog** - Rozszerzony interfejs
   - Sekcja edycji nazwy zespołu
   - Sekcja zarządzania członkami z wizualizacją
   - Wyszukiwarka użytkowników z podglądem na żywo
   - Dodawanie członków przez kliknięcie
   - Usuwanie członków przez przycisk X na badge
   - Walidacja: zespół musi mieć przynajmniej jednego członka
   - Responsywny design z przewijaniem
